Vendoring

Bring the repo into another project

`bruvtext` is a normal repo. Vendor it with a submodule, subtree, or a copied folder. Track the vendored source tree normally. Do not hide it behind a blanket gitignore.

Submodule

Use a git submodule

mkdir -p third_party
git submodule add https://github.com/wegfawefgawefg/bruvtext third_party/bruvtext
git submodule update --init --recursive

To update later: 

git -C third_party/bruvtext fetch origin
git -C third_party/bruvtext checkout master
git -C third_party/bruvtext pull --ff-only
git add third_party/bruvtext
git commit -m "Update bruvtext"

Subtree

Use a git subtree

git subtree add \
  --prefix=third_party/bruvtext \
  https://github.com/wegfawefgawefg/bruvtext master --squash

To update later:

git subtree pull \
  --prefix=third_party/bruvtext \
  https://github.com/wegfawefgawefg/bruvtext master --squash

Copied Folder

Use a plain checked-in copy

  1. Create third_party/bruvtext in the host repo.
  2. Copy the repository contents into that folder.
  3. Commit the vendored source as normal code.
  4. Pull updates manually when needed.

Ignore Rules

Ignore only generated outputs

third_party/bruvtext/build/
third_party/bruvtext/artifacts/
third_party/bruvtext/.codex/
The demo fonts and sample text are included so the showcase and benchmark work out of the box. They are not required by the core library API, and they do not matter unless you build or ship the demo.
