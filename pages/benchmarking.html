Benchmarking

Run the stress test and read the numbers

The benchmark exists to expose implementation mistakes and to show where the time is actually going. It is harsher than a normal game HUD or menu scene.

CLI

Run it without clicking around

./build/debug/demo/bruvtext-demo \
  --benchmark \
  --benchmark-layout scroller \
  --benchmark-runs 800 \
  --benchmark-duration 2

The demo writes CSV samples to artifacts/benchmark/benchmark_log.csv.

Modes

What the layouts mean

  • stacked: dense overlap and heavy alpha overdraw.
  • scroller: lower overlap, better for backend throughput checks.

Interpretation

What to look at

  • CPU/prepare ms: shaping, queue processing, draw-data generation.
  • GPU/frame ms: upload and draw cost in the renderer.
  • Batches: actual renderer-submitted batches, not just library pre-batches.
  • Atlas pages: atlas memory footprint and page churn.
Current measurements show that the GPU path is in decent shape and CPU preparation is still the main cost in the heavier stress cases.

Reality Check

Normal projects are usually lighter

Most games and tools do not have every language active at once. They usually render one active language and a small number of fonts. The benchmark intentionally abuses the system more than a normal UI scene.
